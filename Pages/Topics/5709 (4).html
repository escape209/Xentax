ultimaespio wrote:So in general, would it be possible to make like a tool or maxscript to get the data from any savestate? This is quite interesting.
The only slight problem would be that some pointers to data would now be loaded meaning that say a model begins at offset 12345678 then a pointer say to vertex data might be 123456790, instead of just 18 like it would be in the raw file.
Also this method is really not good for modern computer games because they allocate memory more dynamically, meaning that parts of the same mesh file might be spread in memory over 100 MB apart from each other, and computer games usually strip headers from files when they are loaded in memory, while PS2 games tend to keeps file headers.
When looking at savesstates the same mesh might be in memory several times, with one instance being read-only and the other instances being computed on for animations.

Post by szfzafa » Sun Feb 27, 2011 12:01 pm

Years ago I tried 3DR and GA, got nothing and quit. It's really nice to see so many people made progress here.


To summarize, I'm writing this post.


# Ripping with 3rdparty tools:

Neither 3DR(short for 3D Ripper DX) nor GA(GameAssassin) can do this.
GA can rip but seems got nothing, for nothing in its 3DView, neither does that imported in 3dsMAX.
3DR got plane meshes, completely in a plane, missing Z-depth.
As for OGLE, not tried yet, which's nonsense.

Only 3DVia can rip and get flat meshes but it does have Z-depth(using ZeroGS).
Problem:
1. The scene looks odd perhaps in a view of perspective projection(just as SoftIce says).
2. Half the faces you get are cullfaces, which is inside out.
3. UV mapping. UV might be destroyed, though it looks right in 3dsMax; textures are 4096x16(or 8192x1), in wrong width & height.
Solution:
1. Non-linear transformation for the whole scene, from a perspective projection to a parallel one. But the algorithm REALLY SUCKS for me.
2. Make all faces double sides.
3. ??? not sure yet :'(


# Dumping with motified GS plugin:

GsDx

In the lastest version you can simply press Shift+F8 to dump the current scene, along with a normal snapshot u'll get an .obj & a .gs dump. (I got to know this feather cuz I took a glance at the GsDx src)
Problem:
1. The scene looks odd perhaps in a view of perspective projection.
2. Faces you cannot see in your current view will NOT be dumped, maybe not even rendered, which is different from ZeroGS.
3. As for UV mapping & textures, what's .gs format?
Solution:
1. Non-linear transformation for the whole scene, from a perspective projection to a parallel one. But the algorithm REALLY SUCKS for me.
2. Motify the code, to draw every face in scene and don't consider covering up. (But I haven't got DX11SDK installed yet)
3. ??? (I'm not sure, hoping this won't be a problem

ZeroGS

So far, I find no dumping model functions in ZeroGS src. But ZeroGS perheps draws every face in view without considering the covering up stuff. So you can get a full rip of a person, with his back.
I cannot compile ZeroGS yet, for missing Zlib and sth. But there must be at least one problem:
Problem:
1. The scene looks odd perhaps in a view of perspective projection.
2. ...
Solution:
1. Non-linear transformation for the whole scene, from a perspective projection to a parallel one. But the algorithm REALLY SUCKS for me.
2. ...
(TO DO: add up here;)


Extracting from ISO image/memory
Analysising/Reversing engineering formats

(quoted from invisghost's)
Furry fan and chrrox are amazing at reverse engineering formats. I know furry fan uses cheat engine to look at the memory of a program and get models from that (Although that seems like a lot of work). chrrox seems to do it the old fashion way and just figure out the format by looking at several files of the same type and looking for similarities. Just trying to say that you can have success with almost any method but you must be motivated and willing to learn very complicated systems.
Furryfan's method can avoid file encryption/compression, and may get raw model data (charactors in T-position).
But the only shortcomings - it's not applicable for everyone and every game, since each game/producer has its/his own style, and it's really hard for those unfamiliar with all sorts of 3D model variable styles and structures or lacking experience. I really want an alternative method applicable for ALL PS2 emulatable games XD ^^





PS:
I'm merely a newbie, but with Emulator open source and so many works done by others we can do more!

I know many of your have longlast will and sharp minds, plus PCSX2 & its GS plugins are open source!!
So why wouldn't you guys look into the case?! i.e. motifying the GS plugin code, do the whole-space transformation, etc. I think they are feasible schemes to this subject.
